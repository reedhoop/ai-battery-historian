## 修复 CSP 测试失败（csp_test setUpPage FAILED / strict-dynamic 拦截 inline）
- 根因：`setup.go` 用 `--js .../closure/goog/**/*.js` 把整个 closure 库打进生产包，
  `goog.testing/csp_test.js` 顶层代码无条件注入 strict-dynamic 的 CSP `<meta>`，
  拦截页面无 nonce 内联脚本（base.html line 59 主题切换按钮），并让 closure 测试框架
  在首个测试前检测到 CSP 违规，导致 4 个 csp_test 用例 setUpPage 全 FAIL。
- 修复 `setup.go`：
  1. 收集 closure/goog 输入时排除 `*_test.js`（含 csp_test.js）与 `demos/`，
     **保留整个 testing/ 目录**（否则顺势剔除 goog.testing.TestCase 会引发
     "namespace not provided" 编译错误——TestCase 仅被 `_test.js` 链条引用，属可剪枝依赖）。
  2. 改用 Java `@argfile` 传参，绕开 Windows 命令行长度上限（约 32K，此前报
     "The filename or extension is too long"）。argfile 内反斜杠转成 `/`，避免 Java 转义误解析。
  3. 补回此前误删的 `-jar` 编译选项（上一轮修多个 `...` 展开时漏掉了）。
- 验证：`go run setup.go` 成功（SETUP_EXIT=0）；新 bundle 463623B，`node --check` 通过，
  CSP meta 注入 0、csp_test/doCspViolation 0、app 入口 historian.upload 在、测试框架已剪枝；
  实机启动 `/` 与 `/compiled/historian-optimized.js` 均 200，line 59 内联脚本存在且不再被 CSP 拦截。
- 提交 9969f0a，按惯例推送 origin/master。临时目录 `_tmp_verify`（含手动 strip 备份/脚本/服务二进制）已清理。

## 用户追问：页面状态信息（pass/fail）为什么没了？
- 澄清：页面上显示的测试状态**不是**应用自身单元测试，而是 Closure 库自带的 `goog.testing.csp_test.js`（被原始 `closure/goog/**/*.js` 全量 glob 意外打进生产包）。用户失败报告"4 of 4 tests, 0 passed, 4 failed"正好 4 个、全是 csp 用例，证明页面实际只跑了这一个库测试。
- 应用自己的 `js/*_test.js` 在 optimized 模式下不会加载（不在 historian.upload 依赖树，被 PRUNE）。所以用户记得的"正常 pass"就是 csp_test 通过时的输出；"fail"是其注入 CSP 后自误报。
- 结论：移除 csp_test.js 后页面无任何测试运行、状态栏为空，是符合预期的（不再把别人库测试塞进生产页）。不要在 optimized 包里把 csp_test 加回来。若用户想要健康检查指示，应另做状态面板或仅 dev 模式跑测试，而非复用 csp_test。
