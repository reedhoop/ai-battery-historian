# 通用 Section 分级提取方案 — 洞察报告

> 状态:洞察 / 设计阶段(尚未实现)。生成于 2026-07-19。
> 结论前置:**可行**。但严格限定范围——**Google 上游 Battery Historian 的原始 battery 信息提取一律冻结不动;仅本项目"自己扩展 + 未来新增"的 dumpsys 类功能才纳入本方案。**

---

## 1. 背景与动机

当前代码库里,从一份 txt bugreport 中"切段"的逻辑散落、重复:至少 **7 处**各自维护一个 `inSection` 布尔状态、各自写"逐行扫 + 匹配正则 + 收集到下一个分隔符"的循环。而识别段头用的 3 个正则(`BugReportSectionRE` / `ServiceDumpRE` / `ActivitySubsectionRE`)分散在 `bugreportutils` 与 `historianutils` 两个包里,**没有任何统一的"段"数据模型**。

痛点:
- 同一套"分级分隔"语义被重新实现 7 次,维护成本高(任一 AOSP 格式微调都要改多处)。
- 没有树模型,各 parser 只能"找到某段的起点,扫到 EOF 或下一个匹配"——嵌套关系(章>服务>子段)是隐式的、靠约定。
- 边界处理(优先级前缀、闭合破折号个数、同名多段)在各处各自处理,易不一致。

目标:把这些散落的循环收敛成一个**配置驱动、单遍扫描、产出 Section 树**的通用引擎。

## 3. 方案设计(配置驱动分级解析器)

把分级规则**数据化**,不再写死在循环里:

```go
type Level struct {
    Name        string         // "chapter" | "service" | "activity-sub"
    Start       *regexp.Regexp // 该级分隔端头
    CaptureName string         // 命名分组名:"section" / "service" / "subsection"
}
type Section struct {
    Level    string
    Name     string
    StartLn  int
    EndLn    int
    Lines    []string
    Children []*Section
}
type Config struct{ Levels []Level } // 由外到内:L1 > L2 > L3
```

AOSP 三件套直接映射成 `Level` 数据:
- **L1 章**:`BugReportSectionRE`(`------ X ------`,capture `section`)
- **L2 服务**:`ServiceDumpRE`(`DUMP OF SERVICE [优先级] svc:`,capture `service`)
- **L3 activity 子段**:`ActivitySubsectionRE`(`ACTIVITY MANAGER … (dumpsys activity x)`,capture `subsection`)

**算法(单遍 + 栈)**:bugreport 无显式"结束分隔符",一段天然结束于同级或上级分隔符出现时。

```
stack = [root]
for i, line := range lines:
    for k := len(cfg.Levels)-1; k >= 0; k--:       // 从最内层往外试
        if m := cfg.Levels[k].Start.FindStringSubmatch(line); m != nil:
            name := group(m, cfg.Levels[k].CaptureName)
            popAndFinalize(stack, k)               // 关闭所有 level >= k 的已开段
            node := &Section{Level, name, StartLn:i}
            top(stack).Children = append(..., node)
            stack = append(stack, node)
            break
    // 非分隔行:append 到栈顶段 Lines
```

复杂度 `O(行数 × 级数)`,一次扫描出整棵树。

**查询 API**(替换那 7 个 ad-hoc 函数):

```go
tree := sectionparser.Parse(cfg, reportLines)
tree.Get("service", "power")      // → L2 power(含 CRITICAL 前缀)
tree.Get("chapter", "DUMPSYS")    // → L1
tree.Get("activity-sub", "lmk")   // → activity 树下的 L3
tree.Walk(func(s *Section))       // 遍历回调
sec.Content()                     // 该段文本
```

正则与 AOSP 对齐(已用本机 WSL 的 AOSP 17 源码逐行验证,非仅网页):
- L1 ↔ `dumpstate.cpp:1388` `RunDumpsysText` 输出 `------ %s (/system/bin/dumpsys) ------`
- L2 ↔ `dumpsys.cpp:487` `writeDumpHeader` + `ConvertBitmaskToPriorityType`(返回 `CRITICAL`/`HIGH`/`NORMAL`)
- L3 ↔ `ActivityManagerService.java:11812+` `ACTIVITY MANAGER … (dumpsys activity x)`
- 仓库自带 `_aosp_review/aosp17_samples/` 真实样本交叉验证全部命中。

## 4. 适用范围边界(核心约束)

**❄️ 冻结区(Google 上游,绝对不动):**
- `ExtractBatterystatsCheckin` + checkin/CSV 管线(`parseutils` / `checkinparse` / `checkindelta`)——电池核心提取
- `extractSensorInfo`(sensorservice)——Google 传感器分析
- `IsBugReport` 检测
- `activity/activity.go`(事件日志 CSV)、`dmesg/dmesg.go`(内核 dmesg CSV)等上游分析器
- 其它任何上游 Google 分析器

**✅ 纳入区(本项目扩展 + 未来 dumpsys 功能):**
- `ExtractServiceDump`(helper,本项目)
- `splitSubsections`(本项目,FR-22)
- `packageutils`(本项目,OEM)
- `wearable`(本项目)
- 未来 P4+ 基于 dumpsys 段的新功能(`power` / `alarm` / `procstats` / `activity` 已属本项目,且都经 `ExtractServiceDump` 取段 → 迁移 `ExtractServiceDump` 即自然覆盖)

**实现形态**:新增独立包 `sectionparser`(本项目代码),仅被纳入区调用;Google 代码继续调原函,互不影响。

## 5. 迁移策略(加法,非破坏)

1. **新增** `sectionparser` 包(纯新增,零侵入),复用 3 个正则作为 `Level` 数据,不自带正则。
2. **加单测**,用 `_aosp_review/aosp17_samples/` 真实样本做 golden 断言(顺带把下列边界写进测试):
   - 闭合 5 vs 6 破折号容差(现有 `BugReportSectionRE` 闭合只写 5 个,靠 `\s+` 容差)
   - `CRITICAL/HIGH/NORMAL` 前缀对齐(早期 `\s+` 写进可选组内部的 bug 已修)
   - 同名多段只取首个(与现 `ExtractServiceDump` 行为一致)
3. **逐个迁移**纳入区调用点,每迁一个跑 `go build ./...` + 现有 analyzer 单测,保证输出字节级不变。
4. Google 冻结区保持原样,不触碰。

---

## 6. 边界 / 风险 / 未决项

- `ExtractServiceDump` 与 `extractSensorInfo` 同处 `bugreportutils.go`:本方案**只替换前者**,后者冻结——实施时须精确到函数级。
- `dmesg` / `activity` 是 Google 上游,其跳段逻辑**不在**本方案内。
- **同名多段合并**:AOSP 现在把 `activity`/`power` 等既放 `CRITICAL` 又放无前缀段。通用方案默认"同名多段各自独立节点、调用方按需合并",与现 `ExtractServiceDump` 一致;若希望引擎层自动合并,可在 `Get` 加 merge 选项(待定)。
- 极老 bugreport(无优先级前缀、旧 dumpstate 段头格式)可能失配,但 historian 面向现代报告,可接受。
